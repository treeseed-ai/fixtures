The fixture still has real research questions, but they are implementation-facing rather than speculative. The main question is always some version of:

How do we keep TreeSeed easy to understand without making the example site too thin to catch real regressions?

That leads to smaller questions about content modeling, navigation, docs authoring, agent visibility, forms defaults, and release verification. The [questions](/questions/) index captures those explicitly so the fixture models the same inquiry surface it is meant to teach.

The practical rule is simple: if a contributor cannot infer how to work with TreeSeed from this site, the fixture is not doing enough.
